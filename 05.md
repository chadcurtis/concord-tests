# CORD-05: Invites

How members are handed the keys that make them members, an invite is a shareable URL whose keys live in an encrypted bundle on relays, the link itself carries only a "fetcb token", never the direct keys, so anyone the link reaches can join and nobody who lacks it can, this also means an Invite Link can be revoked without the need to reroll access keys.

----

## 1. The Bundle

An invite delivers a bundle to relays, encrypted by an off-network URL token:
```
CommunityInvite {
  community_id, owner, owner_salt,        // owner proof: community_id == sha256(owner || salt)
  community_root, root_epoch,             // the base access key + its current epoch
  channels: [{ id, key, epoch, name }],   // the granted Channels
  relays,                                 // where the Community publishes
  name, icon,                             // preview, so a parked invite can render
  expires_at,                             // optional, unix ms: past it, the preview still renders, joining refuses
  creator_npub, label                     // optional attribution, echoed in the joiner's Join (below)
}
```
The inviter's own identity is irrelevant to trust, the `community_id` self-certifies the owner (CORD-02), so a bundle can't smuggle a false owner or a fake key for a real Community, a mismatching bundle (its `owner` and `owner_salt` failing to reproduce the `community_id`) is simply refused.

A bundle is attacker-crafted input reached by following a link, so a client MUST bound it before allocating: reject a bundle carrying more than a sane channel count (Vector's ceiling is 256), and truncate `relays` to the Community's cap (CORD-02 §6). Absent bounds, a hostile link is an unbounded-allocation and connect-storm vector.

Attribution is the exception, by choice: a bundle may name its creator and a human label for the link ("Reddit", "Conf 2026"), and an accepting joiner echoes both in their Guestbook Join (CORD-02 §5), which is what makes per-link usage counters possible, all of it inside the token-encrypted bundle, visible to link-holders alone.

An invite bundle is passive, it waits on relays to be *fetched* rather than being pushed to anyone, so a user only encounters one by actively following a link, and even then the client merely pulls it to preview the Community, nothing joins, subscribes, or announces presence until the user explicitly accepts.

## 2. The Link

An invite is a URL whose `#fragment` carries a random 32-byte **fetch-token**, never the keys, and never sent to any server (the fragment stays client-side). The standard is effectively `$BASE#fragment`, and the base is interchangeable: the same fragment opens on `vectorapp.io/invite#…`, on Soapbox's Armada web base, or on any deeplink redirect domain anyone cares to run (as Matrix does with its many `matrix.to`-alikes), only the fragment is protocol and any client recognizing an invite must respect it verbatim. The token alone derives everything needed to find and open the bundle:
```
bundle_key    = hkdf(token, "concord/invite-key")      // decrypts the bundle
bundle_id     = hkdf(token, "concord/invite-locator")  // the addressable coordinate
bundle_signer = hkdf(token, "concord/invite-signer")   // signs the bundle event
```
The bundle is posted as an addressable event, signed by `bundle_signer` at `bundle_id`, its content the NIP-44-encrypted bundle, so only a token-holder can find it, open it, or reissue it, a fetcher rejects any event at the coordinate not signed by the token's signer (an impostor can't squat the link). `bundle_signer` normalizes to a valid key exactly as any derived scalar (CORD-02, A.3), and squatting buys nothing even posted first: a squatter can't produce the token's signature, a fetcher drops their event unread, and the legitimate holder's post stands on its own, the same signer the link always implied. On the wire, this is one of the few Concord events relays see bare (CORD-02 Appendix B):
```jsonc
{ "kind": 33301, "pubkey": "<bundle_signer's pubkey>",
  "content": "<nip44_encrypt(bundle_key, the §1 bundle as JSON)>",
  "tags": [ ["d", "<bundle_id hex>"], ["vsk", "6"] ] }   // vsk 9: the revocation tombstone (below)
```

Because the coordinate is stable, the link itself never changes, re-posting under the coordinate *refreshes the bundle* it addresses (fresh keys behind the same URL, e.g. after a Rekey), so a link shared once survives every rotation. Retiring a link works the same way: the coordinate is re-posted as a **revocation tombstone** (a sub-kind flag, still token-signed, so creator-only), replacing the bundle so a fetcher finds the grave instead of keys, and unlike a relay deletion (best-effort, quietly ignorable) the tombstone is exactly as durable as the bundle it replaced. Retiring the last live link is what flips a Community back to Private (a Refounding, CORD-06).

Clients may quickly identify whether an Invite Link is still live or not by checking for a tombstone at the addressable bundle.

## 3. The Relay Dictionary

An invite must include the Community's relays for immediate reachability, but inserting each one in the Invite URL inflates it massively, often past what a length-restricted platform will accept and into an eyesore besides, so Concord defines a small, trusted **relay dictionary** every client knows, and a community using these relays (a default community) references them by a *single byte* rather than its full URL.

The stock set (four primaries, two Vector and two Soapbox) is selected by one flag, so the common invite carries zero additional relay bytes:
```
1 = wss://jskitty.com/nostr          (Vector)
2 = wss://asia.vectorapp.io/nostr    (Vector)
3 = wss://relay.ditto.pub            (Soapbox)
4 = wss://relay.dreamith.to          (Soapbox)
```
The fragment encodes `[version][flags][relays?][token:32]` as base64url with no padding. `version` is a format byte that also selects the dictionary generation, it's `3` (a client MAY reject any lower value as a legacy link rather than decode it against the wrong dictionary). The `flags` bit selects the stock set, in which case zero relay bytes follow. Otherwise a `count` byte precedes `count` entries, each a leading byte:
```
1..=254   a dictionary id (the relay is one byte, no literal)
0         a wss-implied literal: [len][host bytes], "wss://" re-prepended on decode
255       a verbatim literal:    [len][full URL bytes], for ws:// or exotic schemes
```
The fragment carries at most **3 bootstrap relays**, it only needs to *find* the bundle, which then carries the Community's authoritative relay set (CORD-02 §6).

None of this is forced, the dictionary is a *default*, not a requirement, a client or user can skip the primaries entirely and encode their own relays inline (the escape bytes above), the stock set simply lets a non-technical user create and join a Community with zero configuration, while anyone wanting full control keeps it.

The dictionary is versioned, so it can grow without breaking older links, and both Vector and Soapbox ship it identically, an invite minted by either client opens in the other.

## 4. The Invite List

A creator's minted links sync across their devices (and their *clients*, two apps can serve one npub) as the **Invite List**, the kind `13303` replaceable (CORD-02, Appendix B), NIP-44-encrypted to self, its content the creator's private bookkeeping:
```jsonc
{ "entries": [
    { "token": "<hex>",              // the link's secret AND its merge key
      "community_id": "<hex>",
      "url": "<the shareable link>",
      "label": "Reddit",             // optional, the link's human name
      "created_at": 1719800000,
      "expires_at": 1722400000 } ],  // optional
  "tombstones": [ { "token": "<hex>", "community_id": "<hex>" } ] }
```
Two copies merge without coordination: the token is the merge key, an entry is immutable once minted, tombstones union, and a tombstone always beats an entry, terminally, so a stale device can never resurrect a revoked link. And because two clients can share this one document, the round-trip discipline applies (CORD-02 §6): preserve what you don't understand.

## 5. The Registry

The Invite List above is the creator's *private* bookkeeping, the **Registry** is its member-facing shadow: every creator of live public links also publishes a Control Plane entity (`vsk 8`, CORD-04) listing them *into the Community*, at a coordinate bound to the creator (CORD-02, A.6), so each creator owns exactly their own list and nobody can forge entries into anyone else's. Its content is the live links' *coordinates* only:
```jsonc
["<bundle_id hex>", "<bundle_id hex>"]   // locators, never tokens or URLs, the secret is never listed
```
Members fold every creator's Registry (honored only while its author holds `CREATE_INVITE`) into one aggregate active-set, and that set is the **Public/Private source of truth**: non-empty means a live link exists somewhere and the Community is Public, empty means Private, and retiring the last live link (§2) is precisely what empties the set and triggers the Refounding (CORD-06). A Registry edit accompanies every mint and every retire, and because it carries locators rather than secrets, members can *see* that links exist without gaining the ability to use one.
