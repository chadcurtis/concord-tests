# CORD-00: Private Channels

`draft` `optional`

Private channels can be created on Nostr between multiple parties by sharing a private key between all participants.

All participants sign NIP-59 giftwrap events with a shared key and publish them to relays. Participants with the key can query the channel by its public key:

```json
{ "kinds": [1059], "authors": ["<channel pubkey>"] }
```

To join a private channel, people share invite links. The invite link contains the private key.

## Channel Event

A channel event is a kind 1059 event that appears to be a regular giftwrap, blending in with other giftwrap traffic. But it's actually not a regular giftwrap.

```json
{
  "id": "<id>",
  "kind": 1059,
  "pubkey": "<channel pubkey>",
  "content": "nip44_encrypt({
    "id": "<id>",
    "kind": 13,
    "pubkey": "<real author's pubkey>",
    "content": "nip44_encrypt({
      "id": "<id>",
      "kind": 9,
      "pubkey": "<real author's pubkey>",
      "content": "Hey chat!",
      "tags": [],
      "created_at": 1686840217,
    })",
    "tags": [],
    "created_at": 1686840217,
    "sig": "<real author's signature>"
  })",
  "tags": [
    ["p", "<ephemeral pubkey>"]
  ],
  "created_at": 1686840217,
  "sig": "<channel signature>"
}
```

Since the events are queried by author, the "p" tag is not actually needed, but it should be included anyway so the event is indistinguishable from other giftwraps. The "p" tag should be an ephemeral pubkey that the client may save if it wants to revoke giftwraps later (relays supporting NIP-59 already support deletions by "p" tag recipients).

- Unlike NIP-59, which uses ephemeral authors and fixed "p" tags, Concord uses fixed authors and ephemeral "p" tags. It's reversed.
- Unlike NIP-59, the giftwrap content is encrypted to the channel key, not to the "p" tagged key
- Unlike NIP-59, `created_at` should not be altered or tweaked.
